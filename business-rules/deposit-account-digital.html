Business Rules: Digital Deposit Account Journey

SYNTHETIC — demo only · Consolidated from server/rules/*.js, server/data/*.js, docs/rules/*.md, and inline logic in client/src/pages/*.jsx · updated after the journey's UX redesign
A note on trustworthiness: this works cleanly here because this repo's rules are already explicit and contained in a handful of small files. A real extractor over a production codebase would need human review before anyone acts on its output, not blind trust — especially anywhere logic is implicit rather than written down, which is exactly what the flagged items below are pointing at.

1. Overview

No validation rule. Displays the selected product name and a bulleted benefits list. Purely informational.

2. Phone number

Phone number must be a 10-digit number (non-digit characters stripped before counting). — server/rules/phone.js

3. Verify code (OTP)

The one-time code is mocked: the only code that ever verifies is 123456. — server/rules/phone.js

4. Profile setup

  • Full name, date of birth, and email are required. — server/rules/profile.js
  • Applicant must be at least 18 years old — the eligibility gate for the entire journey.
  • Email becomes the channel for confirmation/welcome messages later.

5. Verify identity (intro)

No validation rule — a splash screen that sets expectations before the identity form. Captures no data.

6. Identity verification (form)

  • Accepted ID types: Driver's License, Passport, Provincial ID Card, Permanent Resident Card.
  • ID must not be expired. — server/rules/identity.js
  • Identity match requires full name, email, and phone number to all be on file — not name alone.

7. Address info

  • Must be a Canadian residential address — PO boxes rejected. — server/rules/address.js
  • Supported provinces: ON, BC, AB, QC, MB, NS, NB, SK, PE, NL.
  • Postal code must match standard Canadian format.

8. Employment info

  • Fields branch by employment status (Employed/Self-employed, Student, Retired, Unemployed).
  • Annual income is always required and feeds the credit cross-sell eligibility check.

9. Credit card offer (conditional)

Only shown if all hold — screen never appears for non-qualifying applicants:

  • Annual income ≥ $24,000
  • Employment status is Employed or Self-employed
  • Applicant is at least 21 years old

If qualifying, displayed credit limit = 20% of annual income, capped at $15,000, rounded to the nearest $100. — server/rules/creditOffer.js

10. Card added (conditional)

No validation rule — an animated transition shown only if the offer was accepted, then auto-advances after ~2 seconds.

11. Tax questions

Captures two declarations as data only. No server call from this step at all, unlike every other step.

12. Account agreements

  • Collapsible agreement cards: chequing (always, expanded) and credit card (only if the offer was accepted, collapsed).
  • Each visible card has its own checkbox; all visible cards must be checked to continue.
Flagged: unlike every other step, this one has no server-side validation call at all — consent is enforced purely client-side. Nothing server-side stops a bypassed request from reaching Fund account.

13. Fund account

  • Presented as selectable option cards, then a deposit amount field.
  • Digital channel offers transfer, Interac e-Transfer, or cheque deposit. — server/rules/funding.js
  • Minimum initial deposit is $25.

14. Confirmation

  • Records the application and returns a generated account number. — server/data/mockDb.js
  • If the credit card offer was accepted, a separate account number is generated for that product.
  • Shows current balance and, if a card was added, its delivery timing (5-7 business days).
Flagged: the second, credit-card-specific account number (cardAccountNumber in server/data/mockDb.js) is generated whenever the offer was accepted — this behavior has no mirror in docs/rules/credit-cross-sell.md or anywhere else.

Summary of flags

  • Account agreements consent is enforced client-side only — no server-side check backs it up. (Carried forward — still true.)
  • Confirmation's second account number for an accepted credit card exists only in code, no doc mirror. (New.)
Two flags from the previous version of this file are now closed: docs/rules/phone-otp.md and docs/rules/fund-account.md both exist and are accurate as of this pass.
