Impact Brief: Joint-Account Support — Digital Deposit Account Journey (Async Invite Model)

SYNTHETIC — demo only · Updated against the current 14-step digital journey after its UX redesign · UX pattern provided directly by the product owner

Proposed change

Two applicants open one joint account, but not in one sitting. User #1 completes the existing digital journey largely as-is, indicating partway through that this is a joint account and providing User #2's name and email. Once User #1 finishes, User #2 gets an emailed invite link into a separate, shorter flow to complete their own required steps. The account is pending until both are done.

Bottom line

Still true from the original analysis: the hard part isn't redesigning existing steps to hold two applicants' data at once — nearly every step stays untouched for User #1. What's changed since the last version of this brief: (1) Confirmation is now a much richer screen, so the "invite sent" messaging needs to fit into a multi-section layout instead of one sentence; (2) Terms & Conditions is now per-product agreement cards, raising a genuinely new question about a joint ownership agreement card; (3) this repo now has a working reference implementation of asynchronous device-linked identity verification — built for branch — which directly informs User #2's identity-verification step here.

Steps affected — User #1's journey

Needs a change

  • Overview — same as before: a new capture point (joint-account toggle + User #2's name and email).
  • Confirmation — now shows a card visual, a products-opened list, a balance box, and conditional card-delivery messaging, not a single paragraph. The "invite sent" messaging needs a place in this expanded layout — real UI work now, not just a sentence.

Unchanged

  • Phone number, Verify code, Profile setup, Verify identity (intro), Identity verification, Address info, Employment info, Credit card offer, Card added, Tax questions, Account agreements, Fund account — all unchanged for User #1. Several look different than in the last version of this brief (phone/OTP and identity verification are now two screens each; "Card added" didn't exist before; funding is now option cards; account agreements are now an accordion) — none of that changes the conclusion.

New: User #2's journey

  • New entry point, authenticated by an invite token.
  • Reuses existing modules: DOB, phone number, address, employment info.
  • Reuses as-is: Identity verification, Tax questions, Terms & Conditions.
  • Explicitly skipped: Credit card offer, Card added, Fund account.
  • New confirmation variant.
New since the last version: the branch journey now has a real, working remote identity-verification mechanism — a server-backed session, a token-linked device, polling for completion. User #2's identity-verification step is structurally similar and could extend that pattern rather than being built from scratch. They're not identical — branch assumes same-visit, real-time completion; this flow may complete hours or days later. Worth deciding explicitly whether these share one mechanism or stay separate.

New: cross-cutting system needs

  • Pending joint-completion state — still nothing in the data model represents this.
  • Two new notification triggers — still no notification/event system exists.
  • Invite token — securely links User #2's session to User #1's application.

Same real-world engineering scope as before — a working reference pattern reduces design risk and uncertainty, not the actual production work (real email delivery, durable persistence, multi-day token lifetime are all still genuinely new).

Channels/segments impacted

  • Scoped to digital, per direction from the product owner.
  • Sharper question than before: should this invite model and branch's remote-ID-V mechanism be unified into one shared capability, or are they legitimately different problems? This question didn't exist when only one of the two patterns existed in the repo.

Open questions for the business

  • Invite token lifetime — how long is the link valid, and what happens if it expires first?
  • Non-completion handling — indefinite pending, revert to single-owner, or User #1 cancels/resends?
  • Age gate for User #2 — same 18+ minimum, or could a minor be a secondary holder?
  • Phone verification for User #2 — full OTP verification, or just captured as data?
  • Ownership/liability model — full/equal rights immediately, or an interim restricted status?
  • Identity cross-check — must User #2's identity match the invite, or can they correct it?
  • Invite management — can User #1 cancel or resend?
  • Shared mechanism with branchNEW — reuse/extend branch's remote-ID-V pattern, or build separately?
  • Joint ownership agreement cardNEW — does a joint account need its own agreement card in the new T&C accordion, for User #2 to accept?
